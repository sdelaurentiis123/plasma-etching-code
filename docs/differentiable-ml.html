Differentiability and Learned Operators

The two claims nobody else ships: gradients through the whole simulator, and neural surrogates for its expensive pieces.

1. Why differentiable at all

A forward simulator answers "given this recipe, what shape results?" A differentiable simulator also answers "how does the shape change if I nudge each recipe knob?" — it provides the gradient \(\partial(\text{profile})/\partial(\text{parameters})\). That unlocks three things:

CapabilityWhat it does
Inverse designspecify a target profile, optimize the recipe (pressures, energies, time) by gradient descent to hit it. "Target shape → process recipe," automatically.
Self-calibrationfit model parameters to reference data by minimizing a loss — this is how you delete rate_scale: instead of one hand-tuned global knob, gradient-fit the physical parameters to ViennaPS/experiment.
Sensitivity analysisrank which knobs the final CD/bow/depth are most sensitive to — process-window and robustness studies for free.

2. The substrate: NVIDIA Warp

Warp is an auto-differentiable Python framework that compiles kernels to CPU or CUDA and automatically generates the backward (adjoint) version of every kernel. So the same wp.kernel that traces rays forward also backpropagates gradients, and wp.mesh_query_ray gives BVH/RT-core traversal. One substrate provides ray tracing and autodiff — which is exactly why it was chosen over raw OptiX (max rays/sec but no autodiff) or wrapping ViennaPS (least owned IP).

Develop on CPU, deploy on GPU

Warp's CPU backend runs on this Apple-silicon dev box, so kernels and their gradients can be written and validated locally now; the identical kernels run on RT cores when the NVIDIA hardware is available. The autodiff is verified the same way on both.

3. What is hard to differentiate, and how to fix it

Three pieces of the simulator each need care:

(a) The radiosity solve — easy

The neutral equilibrium \(\Gamma=(I-(1-s)K)^{-1}\Gamma_{\rm direct}\) is a linear solve, differentiated cleanly by the adjoint method: one extra solve with the transposed operator yields all gradients. This is one more reason to prefer radiosity over stochastic bounce chains, which are noisy to differentiate.

(b) Monte-Carlo noise — reduce it first

Gradients of a noisy estimator are even noisier than the estimator. This is why variance reduction is a prerequisite, not a nicety: QMC, control variates, and radiosity all smooth the flux field so its gradient is usable.

(c) Visibility discontinuities — the real subtlety

Borrowed from differentiable rendering

The visibility term \(V(\mathbf{x},\boldsymbol\omega)\) is a step function — it jumps from 1 to 0 at a silhouette edge. Naïve autodiff sees gradient 0 almost everywhere and \(\infty\) at the jump, missing the fact that moving the surface moves the shadow boundary. Differentiable rendering solved this with edge sampling (Li et al.) and reparameterized integrals: add a boundary term that captures how occlusion edges move with the parameters. The same machinery applies here — the etch front is the silhouette. This is a known, solved class of problem, but it must be implemented deliberately, not assumed.

4. Learned operators: surrogate the expensive physics

Two pieces dominate cost and stiffness: the surface chemistry source term (stiff coverage ODEs) and the multiple-scattering re-emission integral. The genuinely novel move is to replace each with a learned operator — a neural network trained to map inputs to outputs of that sub-problem.

ArchitectureMapsFit for
DeepONetfunction → function via branch/trunk netslocal operators (chemistry: flux+coverage → rate)
Fourier Neural Operator (FNO)field → field in spectral space, resolution-invariantnonlocal operators (re-emission: geometry+direct flux → equilibrium flux)
Why this is "acceleration without loss of fidelity"

You pay the fidelity cost once, at training time, against high-ray-count ground truth from the (now validated) forward simulator. At inference the operator is cheap and noise-free — it never re-samples, so there is no \(1/\sqrt N\) roughness at all. And because it is a neural network, it is differentiable by construction, feeding straight back into inverse design.

5. The foundation-model convergence

The surrogate machinery here — operators that learn the stiff plasma-surface chemistry from simulation data — is the same machinery as a plasma-chemistry foundation model. The etch simulator and the foundation model converge into one codebase: the simulator generates training data and validates the operator; the operator accelerates the simulator. This is the long-term thesis.

6. Honest scoping

What's proven vs. open

Neural operators (FNO/DeepONet) are demonstrated for plasma edge simulations, MHD, and PIC Poisson solvers — so the architecture class is real and works on plasma data. They are not yet demonstrated for feature-scale etch profile surrogates. That gap is the contribution, and the risk: it must be measured against the high-ray baseline on the same Chamfer/ARDE benchmark, or "fast surrogate" is just "fast wrong answer." Do this last, after the forward simulator is validated — see the phase plan.
