petch — State of the Project

A fast, differentiable, GPU-accelerated 3D plasma-etch simulator. Where it stands, what it does, and what it's for. Updated June 2026.

TL;DR — what we're doing

petch predicts how a single sub-micron chip feature — a contact hole, a trench, a 3D-NAND channel — changes shape as a plasma etches it into silicon. It is ~8–15× faster than ViennaPS (the open-source state of the art) on the same GPU — petch is ~1 s and GPU-resident while ViennaPS's level set is CPU-bound, so the ratio is CPU-dependent (~8× when ViennaPS gets a fast many-core CPU, more on a typical one) — just as accurate (it implements every ViennaPS mechanism), and uniquely differentiable — you can take gradients through the whole physics, which lets you inverse-design a recipe from a target shape. It is, as far as we can find, the only open-source GPU-accelerated, differentiable feature-scale etch simulator that exists.

1. What is sub-micron etching?

Making a microchip means carving billions of nanometre-scale 3D structures into silicon. The dominant way to carve them is plasma (dry) etching: a low-pressure gas (e.g. SF₆/O₂) is turned into a plasma, which produces two things that rain down on the wafer —

  • ions — electrically accelerated straight down, energetic, directional; they do the anisotropic (vertical) cutting and sputtering;
  • neutral radicals — uncharged, arriving from all directions (diffuse); they do the chemical etching and passivation.

A patterned mask protects some areas; everywhere else, the surface recedes. Feature-scale (a.k.a. sub-micron) simulation means modelling one individual feature — narrower than a micron — and the evolving shape of its surface: depth, sidewall angle, bottom profile, bowing. (This is distinct from reactor-scale simulation, which models the whole plasma chamber.)

Why it's hard — and why it matters

As a feature gets deep and narrow (high aspect ratio), the etch slows down and distorts: the deep floor is geometrically shadowed, so fewer ions and neutrals reach it. This shows up as ARDE / RIE-lag (aspect-ratio-dependent etch rate), microtrenching (over-etched corners), and bowing. Predicting these before committing a $10k+ wafer lot is exactly what feature-scale simulators are for: designing masks and recipes, hitting a critical-dimension (CD) target, and avoiding defects.

2. How petch works

The surface is a level set \(\phi\) (a signed-distance field; the surface is \(\phi=0\)). Each time step is a clean pipeline:

make geometry → [ marching cubes → Monte-Carlo flux → SF6/O2 chemistry → advect surface → reinit ] × N steps
  1. Mesh the \(\phi=0\) surface into triangles (marching cubes).
  2. Flux: trace ions (directional) and neutrals (diffuse, multi-bounce) through the evolving cavity by GPU ray tracing on a BVH, with a flux↔coverage fixed point — how much of each species reaches every surface point.
  3. Chemistry: convert flux + ion energy into a local etch velocity (the Belen coupled-coverage SF₆/O₂ model — identical to ViennaPS).
  4. Advect the level set by that velocity (upwind Godunov, with CFL substepping).
  5. Reinit \(\phi\) back to a signed distance (GPU Eikonal solver).

The whole pipeline is written in NVIDIA Warp, so it runs on the GPU and is automatically differentiable in one substrate. On a machine without an NVIDIA GPU the same code falls back to a portable CPU path.

(Earlier in the project, before the GPU campaign, petch was ~2× slower than ViennaPS-GPU; the campaign below flipped that.)

4. Accuracy — we implement ≥ ViennaPS's physics

petch's belen.py reproduces the complete ViennaPS SF6O2Etching model, mechanism for mechanism — so "faster" does not mean "cruder":

5. How we got there — the GPU speedup campaign

The original loop was dominated by CPU host operations. We moved them to the GPU one at a time, validating each stayed accuracy-neutral. The big levers, in order of impact:

  • GPU Eikonal reinit (Jacobi Godunov fast-sweep, Warp) — replaced CPU skfmm; the reinit went from ~25% of the loop to <1% (≈58× on that line).
  • Warm-start coverage — seed each step's flux↔coverage fixed point from the previous step, so it converges in 1 iteration instead of ~4 → 8× fewer neutral ray traces, exact same answer.
  • GPU marching cubes (Warp built-in) — 24× faster than skimage on the meshing op.
  • On-device flux — normalization, smoothing (with GPU normals/weights), and source-ray generation all on the GPU; the warm-start nearest-face lookup via a GPU BVH query instead of a CPU KD-tree.
  • cupy GPU sort for the mesh edge-adjacency (the last host bottleneck).

Crucially, every one of these is an opt-in flag that auto-enables on CUDA and falls back to the portable path on CPU — so the tool is fast on a GPU, "just works" (slower) on a laptop, and stays modular. We deliberately did not fuse the loop into a monolithic CUDA graph: it would have made the fast path NVIDIA-only and the code far harder to extend, for only ~1.5× more on top of the existing speedup.

6. The API

The public interface mirrors ViennaPS's Domain/Process shape, so a ViennaPS user migrates in minutes — but it auto-runs the full fast pipeline:

import petch

dom    = petch.Domain.hole(extent=14, dx=0.25, diameter=6, mask=2, depth=18)
model  = petch.SF6O2()                      # = ViennaPS physics; override any parameter
result = petch.Process(dom, model, duration=3.0).run(steps=40)

result.depth          # µm        result.aspect_ratio
result.mesh           # (verts, faces)        result.wall_time   # ~1.6 s vs ViennaPS ~22 s
result.plot("profile.png")                   # cross-section figure
result.save("etch.vtk")                      # ParaView / ViennaPS-compatible

Three layers: this clean one, a config-object layer (override physics/numerics), and the low-level run_etch_3d / Flags / PAR for full control.

7. Honest limitations

  • As-accurate-as-ViennaPS, not yet validated to real wafers. Matching ViennaPS is not the same as matching a specific fab. The real-wafer gap (surface charging, true Knudsen molecular-flow transport) is research-grade and needs experimental data to calibrate.
  • Surface charging is experimental and off by default. We built the infrastructure and calibrated it against the Hwang–Giapis experiment; the reduced "differential electron/ion shadowing" model over-throttles at every setting, because the real effect is a self-consistent floor potential re-deflecting ions (a PIC-class field solve), not geometric shadowing. We document this honestly rather than ship a model that doesn't match the data.
  • ViennaPS is the only runnable open-source peer; other comparisons are against published experimental curves (Belen 2005, de Boer/Blauw, Gomez 2004, Hoekstra–Kushner).

8. The science this unlocks

The combination fast + differentiable makes experiments possible that ViennaPS can't do (it's not differentiable) and that finite differences make impractical (too many slow runs):

  • Differentiable inverse design ★ — gradient-descend the recipe (ion flux, energy, O₂ fraction, sticking) to hit a target profile (vertical walls, a depth, no microtrench). "Tell me the structure; I'll compute the recipe."
  • Differentiable calibration to SEM data ★ — fit the physical constants to a published wafer cross-section by gradient descent on the profile mismatch. This is the credible path from "ViennaPS-accurate" to "real-wafer-accurate."
  • ARDE / RIE-lag landscapes at scale; global sensitivity (which knob controls sidewall angle / bow / microtrench, read straight off the gradients).
  • Optimal time-varying recipes — differentiable optimal control of a recipe schedule, a principled generalization of Bosch-style alternation.
  • Uncertainty quantification / process-window mapping — Monte-Carlo over input tolerances, cheap once a run is ~1 s.
